单一职责原则

简单来说，就是明确分工合作，减少判断流程。将不同的执行逻辑归纳到相应的集合中。
注意：单一职责同样也适用于方法。一个方法应该尽可能做好一件事情。如果一个方法处理的事情太多，其颗粒度会变得很粗，不利于重用
依赖Java多态特性。

比如说，我有两件事，读取事件和写入事件。如果把这两件事都在一个方法中实现，则会显得复杂。

反例
WriterAndReader.java
class WriterAndReader{ //方法中使用if来判断需要执行那种逻辑代码，使整体代码变得复杂 void WriterAndRead(boolean isWrite){ if(isWrite) { } else {} } } 当然，方法层面好解决，在类、接口层面，需要进行一定的分层。
具体思路为；设计一个统一的父类，父类中一些抽象方法，这些抽象在不同的子类中实现不同的功能。
